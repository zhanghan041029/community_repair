## 一、需求分析

### 1.1 系统功能描述

本组完成的系统是**社区维修系统管理平台**，该系统主要实现以下核心功能：

- **居民端**：支持居民在线提交报修工单、实时查询工单处理进度、取消待派单工单、对已完成工单进行星级评价，打造便捷高效的线上报修体验。
- **维修工端**：支持维修工自主注册账号（经管理员审核后生效）、登录系统查看被指派工单、更新工单处理状态（开始维修/完成维修）、查询个人历史工单记录及居民评价。
- **管理员端**：完成对居民账号、维修工账号的统一管控，支持维修工入驻审核（通过/拒绝）、工单手动派发、全量工单查询与监控、数据统计概览（今日新增、待处理、7天趋势等）等管理操作。管理员工单管理模块还额外实现了**待派单优先排序**、**超时工单浅橙色高亮提醒**、**状态筛选即时生效**、以及**批量派单**等增强功能，提升派单效率与防误操作能力。

该系统能应用于**社区物业管理场景**，替代传统电话报修或手工登记模式，解决报修渠道单一、进度追踪困难、派单效率低下、评价反馈缺失、数据统计困难等突出问题。

### 1.2 各个角色具体功能

**（1）居民角色**：
- 账号注册与登录（选择"居民"身份）
- 提交报修工单（标题、故障描述、地址、联系电话）
- 查看个人工单列表（待处理/已派单/处理中/已完成）
- 取消待派单状态的工单
- 对已完成工单进行评价（1-5星评分 + 文字评价）
- 查看个人统计（工单数量分布、近7天趋势图）
- 管理个人资料（头像上传、信息修改、密码修改）

**（2）维修工角色**：
- 自主注册账号（提交后状态为"待审核"）
- 登录后查看被派工单（待接单、处理中、已完成看板）
- 更新工单状态（开始维修 → 处理中；完成维修 → 填写结果 → 已完成）
- 查看历史工单记录（含居民评价详情）
- **历史工单状态筛选**：支持按工单状态（全部/待派单/已派单/处理中/已完成/已取消）筛选历史记录。
- 管理个人资料

**（3）管理员角色**：
- 登录系统（选择"管理员"身份）
- 数据概览（今日新增、待处理、本月完工、在线维修工、7天趋势图、状态分布）
- 居民管理（查询、删除居民账号）
- 维修工管理（查询、新增、审核通过/拒绝、删除）
- 工单管理（全量查询、派单给维修工、查看详情）
  - **待派单优先排序**：列表默认将 `pending` 状态工单排在最前面，同状态下 `urgent` 紧急工单优先展示。
  - **超时高亮提醒**：对超过 3 天未派单的工单，行背景自动设为浅橙色（`#fff7ed`），并在标题旁显示"超时"标签，便于管理员优先处理。
  - **状态筛选即时生效**：状态筛选、居民姓名、维修工姓名、日期等查询条件改变后即时触发后端查询，无需额外点击搜索。
  - **批量派单**：支持管理员勾选多个待派单工单，一键批量指派给同一个空闲维修工，前端逐个调用派单接口并统计成功/失败数量，提高派单效率。
  - **派单防误操作**：派单弹窗中仅展示 `active` 状态的维修工，且每次派单前弹出二次确认框，避免重复派单或派给不合适的维修工。
- 接收系统通知（新工单提醒、维修工注册提醒）

## 三、接口设计

### 3.1 接口规范说明

- **基础路径**：`/api`
- **鉴权方式**：除登录、注册接口外，所有接口需在请求头携带 `Authorization: Bearer <token>`
- **风格**：遵循 RESTful 设计规范
- **请求方式**：GET 请求不携带请求体，POST/PUT 携带 JSON 请求体，文件上传使用 multipart/form-data
- **统一响应格式**：
  ```json
  { "code": 1, "msg": "success", "data": ... }
  ```
  - `code`：1 表示成功，0 表示失败
  - `msg`：提示信息
  - `data`：返回数据（可为对象、数组或 null）

### 3.2 核心接口示例

> （3-5个体现核心功能的接口，像登录注册这种就不需要，这句话要删掉）

#### 接口1：居民提交报修工单

- **地址**：`POST /api/repairs`
- **请求体**：
  ```json
  {
    "title": "厨房水龙头漏水",
    "description": "水龙头接口处不断滴水",
    "address": "1栋101室",
    "contactPhone": "13800000001"
  }
  ```
- **响应示例**：
  ```json
  { "code": 1, "data": { "repairId": 102 } }
  ```
- **业务逻辑**：创建工单状态为 `pending`，并通知所有管理员。

#### 接口2：管理员派单（含批量派单的前端实现）

- **地址**：`POST /api/repairs/assign`
- **请求体**：
  ```json
  { "repairId": 101, "workerId": 5 }
  ```
- **响应示例**：仅返回 code/msg
- **业务逻辑**：验证工单状态为 pending，验证维修工状态为 active，更新工单为 assigned 状态，记录 assign_time，并通知维修工。
- **批量派单说明**：后端保持单条派单接口不变，前端在 `RepairManage.vue` 中通过表格多选收集多个 `pending` 工单，点击"批量派单"按钮后弹出统一弹窗选择维修工，确认后使用 `for` 循环逐个调用 `/api/repairs/assign`，并统计成功/失败数量。仅 `pending` 状态的工单可被勾选，避免误选已派单或已完成工单。

#### 接口3：维修工更新工单状态

- **地址**：`PUT /api/repairs/status`
- **请求体**：
  ```json
  { "repairId": 101, "status": "completed", "result": "已更换水龙头密封圈，测试不再漏水" }
  ```
- **响应示例**：仅返回 code/msg
- **业务逻辑**：验证工单归属当前维修工；status 可为 processing（开始维修）或 completed（完成维修，必须填写 result）；完成后通知居民。

#### 接口4：居民提交评价

- **地址**：`POST /api/evaluations`
- **请求体**：
  ```json
  { "repairId": 101, "score": 5, "comment": "师傅上门很快，维修技术好，非常满意！" }
  ```
- **响应示例**：仅返回 code/msg
- **业务逻辑**：验证工单属于当前居民、状态为 completed、且尚未评价过；评分范围 1-5；插入评价记录。

#### 接口5：管理员审核维修工

- **地址**：`PUT /api/workers/review`
- **请求体**：
  ```json
  { "workerId": 10, "approved": true }
  ```
  或拒绝：
  ```json
  { "workerId": 10, "approved": false, "rejectReason": "技能描述不清晰" }
  ```
- **响应示例**：仅返回 code/msg
- **业务逻辑**：验证维修工状态为 pending；通过则设为 active 并发送通过通知；拒绝则设为 rejected 并记录原因，发送拒绝通知。

## 四、系统设计与实现

### 4.1 系统设计

**核心业务流程（工单处理全流程）**：

```
居民登录系统 → 进入提交报修页面 → 填写工单信息（标题/描述/地址/电话）
    → 后端创建工单（状态：pending）→ 通知所有管理员
    → 管理员登录 → 进入工单管理
    → 列表按 pending 优先、 urgent 次之排序；超时工单以浅橙色高亮显示
    → 通过状态筛选快速定位待派单工单；勾选多个 pending 工单 → 点击批量派单
    → 选择合适的维修工 → 二次确认 → 工单状态变为 assigned
    → 通知维修工有新工单
    → 维修工登录 → 查看待接单列表 → 点击"开始维修"
    → 工单状态变为 processing
    → 维修完成后 → 填写维修结果 → 点击"完成维修"
    → 工单状态变为 completed → 通知居民
    → 居民登录 → 查看已完成工单 → 点击评价 → 提交星级评分和文字
    → 评价记录存入 evaluation 表
```

**维修工入驻审核流程**：

```
维修工填写注册信息 → 提交注册 → 状态为 pending
    → 通知所有管理员有新维修工待审核
    → 管理员查看维修工列表 → 审核通过/拒绝
    → 维修工收到审核结果通知
```

**核心功能实现**：

1. **认证授权**：JWT + ThreadLocal。请求时 JwtInterceptor 解析 Token 并存入 UserContext；Controller/Service 通过 UserContext 获取当前用户；请求结束后 clear 防止线程池污染。
2. **统一异常处理**：@RestControllerAdvice 全局捕获 BusinessException、参数校验异常和未知异常，统一包装为 Result 返回。
3. **参数校验**：Jakarta Validation 注解（@NotBlank、@NotNull、@Min、@Max）在 DTO 上声明，Controller 用 @Valid 触发校验。
4. **角色权限控制**：Controller 调用 `userService.checkRole()` 校验角色；Service 层校验数据归属权（如"工单是否属于当前居民"）。
5. **通知机制**：工单提交/派单/完成、维修工注册/审核等关键节点，通过 NotificationService 插入通知记录，前端轮询显示。
6. **工单管理增强功能**：
   - **后端支持**：派单接口 `POST /api/repairs/assign` 通过校验工单状态为 `pending` 和维修工状态为 `active`，天然具备防重复派单能力；列表查询 VO 包含 `pendingDays` 字段（由 MyBatis `DATEDIFF` 计算），支持超时天数判断。
   - **前端实现**：`RepairManage.vue` 中使用 `computed` 构造 `sortedRows` 实现待派单优先排序；`isOverdue()` 函数结合 `pendingDays` 和 `createTime` 判断超时；`:row-class-name` 绑定 `getRowClass()` 注入浅橙色样式；表格多选 `el-table-column type="selection" :selectable="checkSelectable"` 仅允许 `pending` 行勾选，配合 `handleBatchAssign()` 完成批量派单。

**核心交互设计**：

- **登录页**：三身份选择框（居民/维修工/管理员）+ 账号密码，不匹配身份后端返回"身份错误"
- **居民看板**：Kanban 三列布局（To Do / On Progress / Done），卡片支持取消和评价操作
- **维修工看板**：三列看板，"开始维修"和"完成维修"按钮，完成时弹窗填写维修结果
- **维修工历史工单**：支持按工单状态筛选，快速查看不同阶段的维修记录
- **管理员仪表盘**：4张统计卡片 + ECharts 折线图（7天趋势）+ 柱状图（状态分布）
- **管理员工单管理**：列表按待派单优先排序；超3天未派单工单浅橙色高亮；状态筛选即时生效；支持批量勾选多个待派单工单并一键批量派单
- **派单防误操作**：弹窗中仅展示 `active` 状态的维修工，派单前进行二次确认
- **通知系统**：顶部铃铛图标每30秒轮询未读数量，点击弹出通知列表，支持单条已读和全部已读
