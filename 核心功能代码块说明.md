## 一、基础设施层 — 如何封装通用能力

### 1.1 认证鉴权链路：拦截器 + ThreadLocal + JWT

**设计思路**：将"当前登录用户是谁"这个问题从业务代码中剥离，通过 Spring MVC 拦截器在请求进入 Controller 之前完成认证，并通过 ThreadLocal 传递用户身份。

**三件套协作流程**：

```
HTTP请求 → JwtInterceptor(提取token) → JwtUtil(解析JWT) → UserContext(存入ThreadLocal) → Controller
请求结束 → JwtInterceptor.afterCompletion → UserContext.clear()
```

#### JwtUtil — 令牌的生成与解析

```
back-demo/src/main/java/com/zhan/backdemo/util/JwtUtil.java
```

封装了两个核心能力：**生成**和**解析**。

- `generateToken(userId, role)`：将 `userId` 和 `role` 打入 JWT 的 claims，设置签发时间和过期时间，用 HMAC-SHA 签名后返回字符串
- `getUserId(token)` / `getRole(token)`：反向解析 token，从 claims 中取出 `userId` 和 `role`
- 密钥和过期时间通过 `@Value` 从 `application.yml` 注入，不硬编码

```java
@Component
public class JwtUtil {
    @Value("${jwt.secret}")   private String secret;      // 签名密钥
    @Value("${jwt.expiration}") private Long expiration;  // 过期毫秒数

    public String generateToken(Long userId, String role) {
        Map<String, Object> claims = new HashMap<>();
        claims.put("userId", userId);
        claims.put("role", role);
        return Jwts.builder()
                .claims(claims)
                .subject(String.valueOf(userId))
                .issuedAt(new Date())
                .expiration(new Date(System.currentTimeMillis() + expiration))
                .signWith(Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8)))
                .compact();
    }

    public Long getUserId(String token) {
        Claims claims = Jwts.parser()
                .verifyWith(Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8)))
                .build().parseSignedClaims(token).getPayload();
        return ((Number) claims.get("userId")).longValue();
    }

    public String getRole(String token) {
        return parseToken(token).get("role", String.class);
    }
}
```

#### JwtInterceptor — 请求入口拦截

```
back-demo/src/main/java/com/zhan/backdemo/interceptor/JwtInterceptor.java
```

实现 `HandlerInterceptor` 接口，覆盖两个关键方法：

- `preHandle`：在请求到达 Controller **之前**执行。从 `Authorization` 头部提取 `Bearer <token>`，调用 JwtUtil 解析出 `userId` 和 `role`，写入 UserContext。解析失败直接抛 `BusinessException`（被 GlobalExceptionHandler 捕获返回 code=0）
- `afterCompletion`：在请求处理完成 **之后**（包括异常情况）执行。调用 `UserContext.clear()` 清除 ThreadLocal，防止内存泄漏
- OPTIONS 预检请求直接放行，不校验 token

```java
@Component
@RequiredArgsConstructor
public class JwtInterceptor implements HandlerInterceptor {
    private final JwtUtil jwtUtil;

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
        if ("OPTIONS".equalsIgnoreCase(request.getMethod())) return true;
        String auth = request.getHeader("Authorization");
        if (!StringUtils.hasText(auth) || !auth.startsWith("Bearer ")) {
            throw new BusinessException("未登录或token无效");
        }
        String token = auth.substring(7);
        UserContext.set(jwtUtil.getUserId(token), jwtUtil.getRole(token));
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response,
                                Object handler, Exception ex) {
        UserContext.clear();
    }
}
```

拦截器的路径注册在 `WebMvcConfig` 中：`/login`、`/users/register`、`/workers/register`、`/uploads/**` 四个路径被排除，其余全部拦截。

#### UserContext — ThreadLocal 用户上下文

```
back-demo/src/main/java/com/zhan/backdemo/context/UserContext.java
```

封装为一个不可实例化的工具类，两个 `ThreadLocal` 变量分别存储 `userId` 和 `role`。

**为什么要用 ThreadLocal？** 一次请求从拦截器到 Controller 到 Service 到 Mapper 都在同一个线程里执行，ThreadLocal 保证线程隔离的同时，让整个调用链的任何位置都能无参获取当前用户信息。

```java
public final class UserContext {
    private static final ThreadLocal<Long> USER_ID = new ThreadLocal<>();
    private static final ThreadLocal<String> ROLE = new ThreadLocal<>();

    public static void set(Long userId, String role) { USER_ID.set(userId); ROLE.set(role); }
    public static Long getUserId() { return USER_ID.get(); }
    public static String getRole()   { return ROLE.get(); }
    public static void clear()       { USER_ID.remove(); ROLE.remove(); }
}
```

**安全意义**：Controller/Service 中的业务方法从 `UserContext.getUserId()` 获取当前用户，而非从请求参数。这杜绝了"用户 A 传参 userId=B 来操作 B 的数据"这种水平越权攻击。

### 1.2 统一响应封装：Result + GlobalExceptionHandler

**设计思路**：后端所有接口返回同一个数据结构，前端只需一套解析逻辑。异常通过 `@RestControllerAdvice` 统一拦截，不在 Controller 中写 try-catch。

#### Result<T> — 响应体范型封装

```
back-demo/src/main/java/com/zhan/backdemo/common/Result.java
```

```java
@Data
public class Result<T> {
    private Integer code;   // 1=成功，0=失败
    private String msg;     // 提示信息
    private T data;         // 业务数据（范型）

    public static <T> Result<T> success(T data) { return new Result<>(1, "success", data); }
    public static <T> Result<T> fail(String msg) { return new Result<>(0, msg, null); }
}
```

**使用方式**：Controller 返回 `Result.success(data)` 或 `Result.fail("错误原因")`，序列化后统一为：
```json
{"code":1, "msg":"success", "data":{...}}   // 成功
{"code":0, "msg":"仅待派单状态的工单可以派单", "data":null}  // 失败
```

前端 Axios 响应拦截器统一检查 `code === 0`，自动弹错误提示。

#### GlobalExceptionHandler — 全局异常拦截

```
back-demo/src/main/java/com/zhan/backdemo/exception/GlobalExceptionHandler.java
```

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    // 业务异常 → code=0，消息透传
    @ExceptionHandler(BusinessException.class)
    public Result<Object> handleBusinessException(BusinessException ex) {
        return Result.fail(ex.getMessage(), ex.getData());
    }

    // @Valid 参数校验失败 → 提取字段错误信息
    @ExceptionHandler({MethodArgumentNotValidException.class, BindException.class})
    public Result<Object> handleValidationException(Exception ex) {
        String msg = ex instanceof MethodArgumentNotValidException ve
            ? ve.getBindingResult().getFieldError().getDefaultMessage()
            : "参数校验失败";
        return Result.fail(msg);
    }

    // 兜底：未知异常 → 打印堆栈并返回友好消息
    @ExceptionHandler(Exception.class)
    public Result<Object> handleException(Exception ex) {
        ex.printStackTrace();
        return Result.fail("系统异常：" + ex.getMessage());
    }
}
```

`BusinessException` 是一个继承 `RuntimeException` 的自定义异常，Service 层遇到业务规则不满足时直接 `throw new BusinessException("仅待派单状态的工单可以派单")`，不用一层层 return 错误码。

**封装价值**：Controller 方法体里只有"校验角色 → 调 Service → 包装成功结果"三行核心逻辑，没有 try-catch。

```java
public void checkRole(String role, String... allowedRoles) {
    for (String allowed : allowedRoles) {
        if (allowed.equals(role)) return;

    throw new BusinessException("无权限访问");
}
```

可变参数设计让调用方只需一行代码声明"这个接口允许哪些角色访问"：

### 2.1 Controller 层：角色校验 + 参数拆分 + 调用 Service

```
back-demo/src/main/java/com/zhan/backdemo/controller/RepairController.java
```

Controller 层只做三件事：**校验角色**、**从 UserContext 提取 userId**、**调用 Service 并包装返回结果**。

#### 工单列表 — 按角色过滤数据的实现

```java
@GetMapping
public Result<PageResult<RepairListVO>> listRepairs(
        @RequestParam(required = false, defaultValue = "1") Integer page,
        @RequestParam(required = false, defaultValue = "10") Integer pageSize,
        @RequestParam(required = false) String title,
        @RequestParam(required = false) String status,
        @RequestParam(required = false) String residentName,
        @RequestParam(required = false) String workerName,
        @RequestParam(required = false) String startTime,
        @RequestParam(required = false) String endTime) {
    return Result.success(repairService.listRepairs(
            UserContext.getRole(),   // 关键：传入当前用户角色
            UserContext.getUserId(), // 关键：传入当前用户ID
            page, pageSize, title, status, residentName, workerName, startTime, endTime
    ));
}
```

**设计关键**：同一个 `/repairs` 接口，**不用三个接口区分角色**。角色信息和用户 ID 传入 Service，在 MyBatis XML 中通过动态 SQL 拼接不同的 WHERE 条件：
- 居民登录 → `WHERE user_id = #{userId}`（只看自己的）
- 维修工登录 → `WHERE worker_id = #{userId}`（只看派给自己的）
- 管理员登录 → 不加用户限制（看全部）

这样前端三端都调同一个接口，后端通过角色自动过滤数据。

#### 创建工单 — 事件通知模式

```java
@PostMapping
public Result<Map<String, Object>> createRepair(@Valid @RequestBody RepairCreateRequest request) {
    userService.checkRole(UserContext.getRole(), Constants.ROLE_RESIDENT);
    Map<String, Object> result = repairService.createRepair(UserContext.getUserId(), request);

    // 创建成功后，通知所有管理员
    Long repairId = ((Number) result.get("repairId")).longValue();
    List<SysUser> admins = sysUserMapper.listByRole(Constants.ROLE_ADMIN);
    for (SysUser admin : admins) {
        notificationService.sendNotification(
            admin.getId(), "新工单提醒",
            "居民提交了新工单：" + request.getTitle() + "（工单号 #" + repairId + "），请及时处理。",
            "repair"
        );
    }
    return Result.success(result);
}
```

通知逻辑放在 Controller 而非 Service，因为"通知"是跨模块的协调逻辑（工单模块 + 通知模块 + 用户模块），不应该是 RepairService 的职责。

#### 状态转换的核心校验逻辑

每个状态变更方法都遵循同一套校验顺序：

```
1. 查询工单 → 不存在抛异常
2. 权限校验 → 不是本人/管理员抛异常  
3. 状态校验 → 当前状态不能执行此操作抛异常
4. 执行更新 → 设置新状态+关联字段（时间戳、维修结果等）
5. 发送通知 → 通知相关方
```

**派单 assignRepair：**

```java
public void assignRepair(RepairAssignRequest request) {
    // 第1步：查工单
    RepairOrder order = repairOrderMapper.findById(request.getRepairId());
    if (order == null) throw new BusinessException("工单不存在");

    // 第3步：状态校验 — 只有 pending 能派单
    if (!Constants.REPAIR_STATUS_PENDING.equals(order.getStatus())) {
        throw new BusinessException("仅待派单状态的工单可以派单");
    }

    // 维修工校验 — 必须是 active 状态
    workerService.getActiveWorker(request.getWorkerId());

    // 第4步：更新工单（一次 UPDATE 同时设 workerId、状态、派单时间）
    RepairOrder update = new RepairOrder();
    update.setId(order.getId());
    update.setWorkerId(request.getWorkerId());
    update.setStatus(Constants.REPAIR_STATUS_ASSIGNED);
    update.setAssignTime(LocalDateTime.now());
    repairOrderMapper.update(update);

    // 第5步：通知维修工
    notificationService.sendNotification(
        request.getWorkerId(), "新派单提醒",
        "您有新的维修工单：" + order.getTitle(), "assigned");
}
```

**状态更新 updateRepairStatus — 多分支校验：**

```java
public void updateRepairStatus(RepairStatusRequest request, Long workerId) {
    RepairOrder order = repairOrderMapper.findById(request.getRepairId());
    if (order == null) throw new BusinessException("工单不存在");

    // 第2步：只能更新指派给自己的工单
    if (!workerId.equals(order.getWorkerId())) {
        throw new BusinessException("只能更新指派给自己的工单");
    }

    String status = request.getStatus();
    // 白名单校验：只接受 processing 或 completed
    if (!"processing".equals(status) && !"completed".equals(status)) {
        throw new BusinessException("状态只能更新为 processing 或 completed");
    }

    // 分支1：设为 processing 必须是 assigned 状态
    if ("processing".equals(status) && !"assigned".equals(order.getStatus())) {
        throw new BusinessException("仅已派单状态的工单可以开始处理");
    }

    // 分支2：设为 completed 必须是 assigned 或 processing 状态
    if ("completed".equals(status)) {
        if (!"processing".equals(order.getStatus()) && !"assigned".equals(order.getStatus())) {
            throw new BusinessException("当前工单状态无法标记为完成");
        }
        if (!StringUtils.hasText(request.getResult())) {
            throw new BusinessException("完成工单时必须填写维修结果");
        }
        update.setResult(request.getResult());
        update.setFinishTime(LocalDateTime.now());
    }

    update.setStatus(status);
    repairOrderMapper.update(update);
    // 通知居民工单进度变更
}
```

#### 所有权校验的两个层次

系统区分"操作权限"和"查看权限"，分别封装为两个私有方法：

```java
// 操作权限 — 修改/取消工单时用
// 规则：管理员可操作一切，居民只能操作自己的工单
private void checkRepairOwner(RepairOrder order, String role, Long userId) {
    if ("admin".equals(role)) return;
    if ("resident".equals(role) && userId.equals(order.getUserId())) return;
    throw new BusinessException("无权限操作该工单");
}

// 查看权限 — 查看详情时用
// 规则：管理员看全部，居民看自己的，维修工看派给自己的
private void checkRepairAccess(RepairDetailVO detail, String role, Long userId) {
    if ("admin".equals(role)) return;
    if ("resident".equals(role) && userId.equals(detail.getUserId())) return;
    if ("worker".equals(role) && userId.equals(detail.getWorkerId())) return;
    throw new BusinessException("无权限查看该工单");
}
```

#### 分页查询 — 角色级数据过滤

```java
public PageResult<RepairListVO> listRepairs(String role, Long userId,
        Integer page, Integer pageSize,
        String title, String status, String residentName, String workerName,
        String startTime, String endTime) {
    int offset = (page - 1) * pageSize;
    // 先查总数（分页组件需要）
    long total = repairOrderMapper.countRepairs(role, userId, title, status,
        residentName, workerName, startTime, endTime);
    // 再查当前页数据
    List<RepairListVO> rows = repairOrderMapper.listRepairs(role, userId, title, status,
        residentName, workerName, startTime, endTime, offset, pageSize);
    return new PageResult<>(total, rows);
}
```

MyBatis XML 中通过 `<if test="role == 'resident'">` 等标签动态拼接 WHERE 条件，实现同一个 SQL 对不同角色返回不同范围的数据。

### 2.3 DTO/VO 设计 — 入参与出参分离

| 类型 | 用途 | 示例 |
|------|------|------|
| `RepairCreateRequest` (DTO) | 创建工单的入参 | title, description, address, image |
| `RepairAssignRequest` (DTO) | 派单的入参 | repairId, workerId |
| `RepairStatusRequest` (DTO) | 状态变更的入参 | repairId, status, result |
| `RepairUpdateRequest` (DTO) | 修改工单的入参 | id, title, description, address |
| `RepairListVO` (VO) | 列表出参 | id, title, status, userName, workerName, createTime, pendingDays |
| `RepairDetailVO` (VO) | 详情出参 | 列表字段 + description, address, image, assignTime, finishTime |

**封装原则**：入参 (DTO) 用 `@Valid` 做字段校验，出参 (VO) 可以包含跨表联查的计算字段（如 `userName` 是 Join `sys_user` 得到的，`pendingDays` 是 MyBatis 中 `DATEDIFF` 计算的）。

## 三、工单管理模块 — 前端实现（重点）

### 3.1 管理员端 — RepairManage.vue

```
front-demo/src/views/admin/RepairManage.vue
```

这是整个前端最复杂的页面，集成了 5 个子功能。下面逐一解释每项功能的封装方式和实现逻辑。

#### 3.1.1 状态筛选 — 即时生效的双向绑定

```html
<!-- 下拉框改变时 → 页码重置为1 → 重新请求后端 -->
<el-select v-model="query.status" placeholder="状态筛选" clearable
           @change="() => { query.page = 1; fetchData() }">
```

实现方式：
- `query` 用 Vue 的 `reactive()` 包裹，`v-model` 实现双向绑定
- 用户选择状态时触发 `@change` 回调：先 `query.page = 1`（防止新筛选结果不足当前页），再 `fetchData()`
- `fetchData()` 将 `query` 中的所有字段传给 `listRepairs()` API，后端 MyBatis 动态拼接 SQL 条件
- 重置按钮把 `query` 所有字段恢复默认值，同样调 `fetchData()`

#### 3.1.2 待派单优先排序 — computed 计算属性

```js
const sortedRows = computed(() => {
  return [...rows.value].sort((a, b) => {
    // pending 排在最前面
    if (a.status === 'pending' && b.status !== 'pending') return -1
    if (a.status !== 'pending' && b.status === 'pending') return 1
    // 同为 pending 时，urgent 优先
    if (a.priority === 'urgent' && b.priority !== 'urgent') return -1
    if (a.priority !== 'urgent' && b.priority === 'urgent') return 1
    return 0
  })
})
```

**为什么用 computed 而不是直接改 rows？** `rows` 是后端返回的原始数据，`sortedRows` 是前端排序后的视图数据。使用 `[...rows.value]` 浅拷贝后再 sort，不污染原始数组。`computed` 自动追踪 `rows` 的变化，数据更新后自动重新排序，表格绑定 `:data="sortedRows"`。

#### 3.1.3 超时工单高亮 — 判定 + CSS 行样式

**判定函数**：
```js
const isOverdue = (order) => {
  // 非 pending 状态无需判断
  if (order.status !== 'pending') return false
  // 优先用后端计算好的 pendingDays 字段
  if (order.pendingDays != null) return order.pendingDays > 3
  // 兜底：前端根据 createTime 计算
  if (order.createTime) {
    return Date.now() - new Date(order.createTime).getTime() > 3 * 24 * 60 * 60 * 1000
  }
  return false
}
```

**行样式注入**：
```js
const getRowClass = ({ row }) => isOverdue(row) ? 'overdue-row' : ''
```

Element Plus 的 `el-table` 提供 `:row-class-name` 属性，每行渲染时调用此函数设 CSS class。

**CSS 深度选择器**：
```css
:deep(.overdue-row > td)       { background-color: #fff7ed !important; }  /* 浅橙 */
:deep(.overdue-row:hover > td) { background-color: #ffedd5 !important; }  /* hover 深一点 */
```

由于 `scoped` 样式无法穿透到 `el-table` 内部生成的 `<td>`，必须用 `:deep()` 穿透。

#### 3.1.4 单条派单 — 弹窗 + 维修工选择器

流程分为三步：打开弹窗 → 选择维修工 → 确认派单。

**第一步：打开弹窗**
```js
async function showAssignDialog(repairId) {
  assignRepairId.value = repairId
  // 从当前列表中找到对应工单（不用再发请求）

  assignVisible.value = true
}
```

## 四、前端基础设施 — 请求/状态/路由如何串联

### 4.1 Axios 拦截器 — 请求-响应管道

```
front-demo/src/utils/request.js
```

通过 `axios.create()` 创建实例后注册两个拦截器：

**请求拦截器（发请求前）**：
```js
request.interceptors.request.use((config) => {
  // 1. 自动从 Pinia store 取 token，附加到 Authorization 头
  if (userStore.token) config.headers.Authorization = `Bearer ${userStore.token}`
  // 2. FormData 上传时删除 Content-Type（让浏览器自动设置 multipart boundary）
  if (config.data instanceof FormData) delete config.headers['Content-Type']
  return config
})
```

**响应拦截器（收到响应后）**：
```js
request.interceptors.response.use(
  (response) => {
    // code=0 是业务错误 → 弹提示 + reject
    if (response.data.code === 0) {
      ElMessage.error(response.data.msg || '请求失败')
      return Promise.reject(new Error(response.data.msg))
    }
    return response.data  // 成功时只返回 data，调用方直接拿业务数据
  },
  (error) => {
    // HTTP 401 → token 过期/无效 → 清空 store + 跳登录
    if (error.response?.status === 401) {
      userStore.logout()
      router.push('/login')
    }
    return Promise.reject(error)
  }
)
```

**封装价值**：业务代码调用 API 时直接 `const res = await listRepairs(params)`，拿到的 `res` 已经是 `{ code:1, data:..., msg:"..." }` 结构。token 附加、错误提示、401 跳转全部由拦截器处理。

### 4.2 Pinia Store — 登录态 + JWT 过期检测

```
front-demo/src/store/user.js
```

采用 Pinia 的 Composition API 风格，核心是 `isLoggedIn` 这个 computed：

```js
// 客户端解析 JWT 的 exp 字段，判断是否过期
function isTokenExpired(token) {
  if (!token) return true
  try {
    const payload = JSON.parse(atob(token.split('.')[1]))  // Base64 解码 JWT payload
    return Date.now() >= payload.exp * 1000
  } catch { return true }
}

export const useUserStore = defineStore('user', () => {
  const token = ref('')
  const isLoggedIn = computed(() => !!token.value && !isTokenExpired(token.value))
  // ...
}, {
  persist: {
    storage: localStorage,
    paths: ['token', 'userId', 'role']  // 仅这三个字段持久化到 localStorage
  }
})
```

`isTokenExpired` 在客户端直接做 JWT 过期校验，**不需要发请求到后端**。这有两个好处：
- 路由守卫 `isLoggedIn` 判断可以同步完成，不阻塞页面跳转
- 即使 JWT 刚过期，前端也能立即检测到，直接跳登录页

### 4.3 路由守卫 — 三层防护

```
front-demo/src/router/index.js
```

```js
router.beforeEach((to, from, next) => {
  const userStore = useUserStore()

  // 第1层：免登录页面直接放行（登录/注册页）
  if (to.meta.noAuth) {
    if (userStore.isLoggedIn && to.name === 'Login') return next('/')  // 已登录无需再登录
    return next()
  }

  // 第2层：未登录 → 跳登录页
  if (!userStore.isLoggedIn) return next('/login')

  // 第3层：角色不匹配 → 跳首页（如居民访问管理员的工单管理页）
  if (to.meta.roles && !to.meta.roles.includes(userStore.role)) return next('/')

  next()
})
```

三层防护由浅入深：先看是否免登录 → 再看是否已登录 → 最后看角色是否匹配。权限声明通过路由的 `meta.roles` 数组完成，如 `meta: { roles: ['admin'] }`。

### 注册时 BCrypt 加密

```java
public Map<String, Object> registerWorker(WorkerRegisterRequest request) {
    // 用户名/手机号唯一性校验
    if (sysUserMapper.findByUsername(request.getUsername()) != null)
        throw new BusinessException("用户名已存在");

    SysUser user = new SysUser();
    user.setPassword(passwordEncoder.encode(request.getPassword())); // BCrypt 加密
    user.setRole("worker");
    user.setStatus("pending");        // 注册后待审核
    user.setRegisterType("register"); // 标记为自注册
    sysUserMapper.insert(user);

    // 通知所有管理员有新注册待审核
    List<SysUser> admins = sysUserMapper.listByRole("admin");
    for (SysUser admin : admins) { notificationService.sendNotification(...); }
    return data;
}
```

`passwordEncoder` 是 Spring Security 的 `BCryptPasswordEncoder`，配置在 `PasswordConfig` 中。密码在入库前经过 BCrypt 哈希，数据库中不存明文。

### 设计原则

| 层次 | 原则 | 体现 |
|------|------|------|
| 前端路由 | 角色级页面隔离 | `meta.roles` + `beforeEach` 守卫 |
| 前端 API | 统一拦截，业务代码无感 | 拦截器自动加 token、处理 code=0、处理 401 |
| 后端 Controller | 只做调度，不做业务 | checkRole → 取 userId → 调 Service → 包装 Result |
| 后端 Service | 所有业务规则集中管理 | 状态流转、所有权校验、通知触发 |
| 后端 Mapper | SQL 与 Java 分离 | MyBatis XML 管理 SQL，动态条件拼接 |
| 数据库 | 统一用户表 + 角色字段 | `sys_user` 一表三用（resident/worker/admin） |

### 数据表关系

```
sys_user（用户表）
  ├─ role: resident / worker / admin
  ├─ status: active / pending / rejected（维修工专用字段）
  │
  └─ 1:N → repair_order（工单表）
              ├─ user_id   → 报修居民（FK → sys_user.id）
              ├─ worker_id → 指派的维修工（FK → sys_user.id）
              ├─ status: pending → assigned → processing → completed
              │                                   └→ cancelled
              └─ 1:1 → evaluation（评价表）
                          ├─ repair_id → 关联的工单
                          ├─ score → 评分(1-5)
                          ├─ comment → 评价内容
                          └─ reply → 维修工回复
```
