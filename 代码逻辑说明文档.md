### 2.2 common/ — 通用工具类

#### Constants.java（常量类）
```
路径：common/Constants.java
```
**核心逻辑**：用 `public static final` 定义系统的所有常量字符串，避免魔法值。
- **角色常量**：`ROLE_RESIDENT = "resident"`、`ROLE_WORKER = "worker"`、`ROLE_ADMIN = "admin"`
- **用户状态常量**：`USER_STATUS_ACTIVE`、`USER_STATUS_PENDING`、`USER_STATUS_REJECTED`
- **注册来源常量**：`REGISTER_TYPE_REGISTER`（自主注册）、`REGISTER_TYPE_ADMIN_ADD`（管理员添加）
- **工单状态常量**：`pending`、`assigned`、`processing`、`completed`、`cancelled`
- 构造方法设为 `private`，防止外部实例化

**为什么这样设计**：
- 避免在代码里到处写字符串 "admin"/"pending"，编译期无法检查拼写错误
- 修改常量值时只需改一个地方，所有引用自动生效

#### Result.java（统一响应封装）
```
路径：common/Result.java
```
**核心逻辑**：泛型类 `Result<T>`，封装所有接口的返回格式。
- 字段：`code`（1=成功，0=失败）、`msg`（提示信息）、`data`（返回数据）
- 静态工厂方法：
  - `Result.success(data)` → 创建 code=1 的结果
  - `Result.fail(msg)` → 创建 code=0 的结果

**为什么这样设计**：
- 前端只要看到 `code` 字段就知道请求是否成功
- 所有接口返回格式统一，前端解析方便
- 全局异常处理器也用它返回错误信息

#### PageResult.java（分页结果封装）
```
路径：common/PageResult.java
```
**核心逻辑**：封装分页查询结果。
- `total`：总记录数
- `rows`：当前页数据列表

### 2.3 config/ — 配置类

#### PasswordConfig.java（密码加密配置）
```
路径：config/PasswordConfig.java
```
**核心逻辑**：向 Spring 容器注册一个 `BCryptPasswordEncoder` Bean。
- `@Configuration`：标记为配置类
- `@Bean`：将该对象交给 Spring 管理

**为什么用 BCrypt**：
- 自动加盐（salt），同样的明文每次加密结果不同
- 抗暴力破解，不可逆
- 验证时用 `passwordEncoder.matches(rawPassword, encodedPassword)` 比对

#### WebConfig.java（Web 全局配置）
```
路径：config/WebConfig.java
```
**核心逻辑**：实现 `WebMvcConfigurer` 接口，配置三样东西：

**① 拦截器注册**：
- 创建 `JwtInterceptor`，拦截所有请求路径 `/**`
- 排除路径：`/login`、`/users/register`、`/workers/register`、`/uploads/**`
- 除登录注册和静态文件外，所有请求都必须经过 JWT 验证

**② CORS 跨域配置**：
- 允许所有来源（`allowedOriginPatterns("*")`）
- 允许所有 HTTP 方法（GET/POST/PUT/DELETE/OPTIONS）
- 允许携带 Cookie（`allowCredentials(true)`）

**原因**：前端（Vite 开发服务器）默认在 3000 端口，后端在 8080 端口，不同端口属于跨域，必须配置 CORS 才能请求。

**③ 静态资源映射**：
- 预留的文件映射配置（头像通过 UploadController 直接读取）

### 2.4 context/ — 用户上下文（ThreadLocal）

#### UserContext.java
```
路径：context/UserContext.java
```
**核心逻辑**：用两个 `ThreadLocal` 变量存储当前请求的用户信息。
- `USER_ID`：当前登录用户的 ID
- `ROLE`：当前登录用户的角色

**生命周期**：
1. **设置**：`JwtInterceptor.preHandle()` 中调用 `UserContext.set(userId, role)` 存入
2. **读取**：Controller 和 Service 中调用 `UserContext.getUserId()` / `UserContext.getRole()` 获取
3. **清除**：`JwtInterceptor.afterCompletion()` 中调用 `UserContext.clear()` 移除

**为什么用 ThreadLocal**：
- Tomcat 为每个 HTTP 请求分配一个线程处理
- ThreadLocal 数据只在当前线程内可见，天然线程安全
- 不需要在每个 Controller 方法参数中传递 userId，简化代码
- 请求结束必须 `clear()`，否则线程池复用时会造成数据污染

### 2.6 exception/ — 异常处理

#### BusinessException.java（业务异常）
```
路径：exception/BusinessException.java
```
**核心逻辑**：继承 `RuntimeException`，额外附带一个 `data` 字段。
- 可以携带附加数据（如登录校验失败时返回用户状态信息）
- 抛出此异常，会被全局异常处理器捕获并返回给前端

#### GlobalExceptionHandler.java（全局异常处理器）
```
路径：exception/GlobalExceptionHandler.java
```
**核心逻辑**：使用 `@RestControllerAdvice` 拦截所有 Controller 抛出的异常。

**三种异常处理**：
1. **BusinessException** → 返回 `Result.fail(异常消息, 附带数据)`
2. **参数校验异常**（`MethodArgumentNotValidException` / `BindException`）→ 提取第一个字段校验错误的提示文本返回前端
3. **兜底异常**（`Exception`）→ 打印堆栈，返回 "系统异常"

**举例**：登录时用户名密码错误，`UserService` 抛出 `BusinessException("用户名或密码错误")`，全局异常处理器将其包装成 `{ code: 0, msg: "用户名或密码错误" }` 返回前端。

### 2.7 interceptor/ — JWT 拦截器

#### JwtInterceptor.java
```
路径：interceptor/JwtInterceptor.java
```
**核心逻辑**（`preHandle` 方法）：
1. 跳过 OPTIONS 预检请求
2. 从请求头 `Authorization` 中提取 `Bearer <token>`
3. 调用 `JwtUtil.parseToken(token)` 验证有效性
4. 调用 `JwtUtil.getUserId(token)` 和 `getRole(token)` 解析出用户ID和角色
5. 存入 `UserContext`
6. 返回 `true` 放行

**`afterCompletion` 方法**：请求结束后清除 `UserContext`。

**作用**：实现无状态的 Token 认证。每次请求携带 JWT，服务端解析后就知道是谁在操作，不需要 Session。

### 2.8 util/ — 工具类

#### JwtUtil.java（JWT 工具类）
```
路径：util/JwtUtil.java
```
**核心逻辑**：
- 从 `application.yml` 读取 `jwt.secret` 和 `jwt.expiration`（7200000ms = 2小时）
- **生成 Token**：`generateToken(userId, role)` — 用 HMAC-SHA256 签名，payload 中存放 userId 和 role
- **解析 Token**：`parseToken(token)` — 验证签名和有效期
- **提取信息**：`getUserId(token)` / `getRole(token)` — 从 payload 中读取 claims

### 2.10 service/ — 业务逻辑层（核心）

#### UserService.java（用户服务）
```
路径：service/UserService.java
```

**login() — 登录逻辑**：
1. 根据 username 查找用户
2. 用 BCrypt 验证密码是否正确
3. **验证选择身份与实际角色是否一致**（新增功能：避免选错身份登录）
4. 如果是维修工，额外检查审核状态（待审核/已拒绝则返回部分信息+错误）
5. 通过后用 `JwtUtil` 生成 Token 并返回

**registerResident() — 居民注册**：
1. 检查用户名和手机号是否已存在
2. 加密密码
3. 插入用户，角色设为 `resident`，状态设为 `active`（直接可用）

**uploadAvatar() — 头像上传**：
1. 校验文件格式（只允许 jpg/png/gif/webp）
2. 校验文件大小（最大 2MB）
3. 按日期建目录：`uploads/avatar/20260617/uuid.png`
4. 更新用户记录中的 avatar 字段
5. 返回相对路径 `/uploads/avatar/20260617/uuid.png`

**changePassword() — 修改密码**：
1. 验证旧密码是否正确
2. 新密码至少6位
3. 加密后更新

**checkRole() — 角色校验工具方法**：
- 传入当前用户角色和允许的角色列表
- 不匹配则抛出异常，实现权限控制

#### WorkerService.java（维修工服务）
```
路径：service/WorkerService.java
```

**registerWorker() — 维修工自主注册**：
1. 创建用户，角色 `worker`，状态 `pending`（待审核）
2. 通知所有管理员："有新维修工注册，请审核"

**reviewWorker() — 管理员审核维修工**：
1. 校验维修工存在且状态为 pending
2. 通过 → 状态改为 `active`，发送通知"恭喜，您的账号已通过审核"
3. 拒绝 → 状态改为 `rejected`，记录拒绝原因，发送通知"很遗憾，您的账号未通过审核"

**deleteWorker() — 删除维修工**：
1. 检查该维修工是否有未完成的工单
2. 有未完成工单则拒绝删除
3. 没有则删除

**listWorkers() — 维修工列表**：
- 管理员查询所有维修工（支持姓名/手机/状态筛选，分页）
- 返回信息包括：工单数、平均评分

---

#### RepairService.java（工单服务 — 核心业务）
```
路径：service/RepairService.java
```

**createRepair() — 提交工单**：
1. 创建工单，状态设为 `pending`（待派单）
2. 设置优先级（priority），默认 `normal`，居民可选择 `urgent`（紧急）
3. 查询所有管理员，给每人发送通知："居民提交了新工单 XXX，请及时处理"

**listRepairs() — 分页查询工单**：
- 根据 role 参数拼接过滤条件：
  - `resident` → 只查自己的（user_id = 当前用户）
  - `worker` → 只查分配给自己的（worker_id = 当前用户）
  - `admin` → 查所有
- 支持按 status、时间范围筛选
- LEFT JOIN 评价表获取评分信息
- **ORDER BY CASE priority WHEN 'urgent' THEN 0 ELSE 1 END**：紧急工单排在前面

**assignRepair() — 管理员派单**：
1. 验证工单状态为 `pending`
2. 验证维修工存在且状态为 `active`
3. 更新工单：worker_id 设为指定工人，状态改为 `assigned`，记录 assignTime
4. 向维修工发送通知："您有一个新工单待处理"

**updateRepairStatus() — 维修工更新工单状态**：
1. 验证工单分配给当前维修工
2. `processing`：从 `assigned` 变为处理中
3. `completed`：从 `assigned` / `processing` 变为已完成，必须填写 repair_result
4. 完成后通知居民："您的工单已处理完成"

**getAdminDashboardStats() — 管理后台数据统计**：
- `todayNew`：今日新增工单数
- `pendingTotal`：待处理总工单数
- `monthCompleted`：本月完成工单数
- `totalWorkers`：活跃维修工人数
- `dailyTrend`：最近7天每天新增工单趋势（通过 countDaily 查询）
- `statusDist`：各状态工单分布
- `recentList`：最近10条工单

#### EvaluationService.java（评价服务）
```
路径：service/EvaluationService.java
```

**createEvaluation() — 提交评价**：
1. 验证该工单属于当前居民
2. 验证工单状态为 `completed`（只有已完成的工单才能评价）
3. 验证尚未评价过（同一工单只能评价一次）
4. 插入评价记录（评分 1-5，可选评价文字）

**replyEvaluation() — 维修工回复评价（新增）**：
1. 根据评价ID查找评价记录
2. 通过工单ID获取工单信息，验证当前维修工是该工单的负责人
3. 更新评价记录的 `reply`（回复内容）和 `replyTime`（回复时间）
4. **给居民发送通知**："维修工已回复您的评价（工单：XXX）：回复内容"

**getEvaluationByRepairId() — 查看评价**：
- 居民只能看自己的评价
- 维修工只能看分配给自己工单的评价
- 管理员可以看所有评价

---

#### NotificationService.java（通知服务）
```
路径：service/NotificationService.java
```

**sendNotification() — 发送通知**：
- 接收 userId（接收者）、title（标题）、content（内容）、type（类型）
- 插入通知记录（isRead = 0 默认未读）
- **通知类型**：`repair`（新工单）、`new_repair`（新报修）、`assigned`（派单）、`processing`（处理中）、`completed`（完成）、`evaluation_reply`（评价回复）、`worker_register`（维修工注册）、`worker_review`（审核结果）、`password_reset`（密码重置）

**countUnread() — 未读数量**：`SELECT COUNT(*) WHERE user_id = ? AND is_read = 0`

**markAllRead() — 全部已读**：`UPDATE SET is_read = 1 WHERE user_id = ?`

---

#### AnnouncementService.java（公告服务 — 新增）

```
路径：service/AnnouncementService.java
```

**create() — 发布公告**：
1. 根据 publisherId 查询发布者名称
2. 创建公告记录（title、content、publisherId、publisherName）
3. **批量通知所有居民和活跃维修工**：
   - 查询所有居民（`listByRole("resident")`）
   - 查询所有活跃维修工（`listActiveWorkers()`）
   - 使用 `NotificationMapper.batchInsert()` 一次插入所有通知记录
   - 通知类型为 `announcement`，内容为"管理员发布了新公告：XXX"
4. 整个过程在一个事务中（`@Transactional`），保证公告和通知的原子性

**listAll() — 公告列表**：按创建时间倒序返回所有公告。

**delete() — 删除公告**：管理员可以删除已发布的公告。

### 2.11 controller/ — 控制器层

**设计原则**：
- 使用 `@RequestAttribute("user")` 或 `UserContext` 获取当前用户
- 角色校验：调用 `userService.checkRole()` 或 `workerRole()`
- 所有请求参数使用 `@Valid` 进行校验
- 返回统一 `Result` 格式

**关键接口对照表**：

|------|------|-------------------|------|

### 3.2 utils/request.js — Axios HTTP 客户端

**核心逻辑**：
- 创建统一的 Axios 实例，`baseURL: ''`（空字符串，配合 Vite 代理转发）

**请求拦截器**（每次请求前执行）：
1. 从 Pinia userStore 中读取 token
2. 如果 token 存在，设置 `Authorization: Bearer <token>` 请求头
3. 如果是 FormData 上传，删除 `Content-Type` 让浏览器自动设置边界

**响应拦截器**（每次收到响应后执行）：
1. 先看 HTTP 状态码是否为 401 → 未登录，清除状态并跳转到登录页
2. 再看业务 `res.data.code === 0` → 业务失败，弹出错误提示并 reject
3. 否则返回 `res.data`（包含 code/msg/data）

**为什么这样设计**：
- 前端所有 API 调用都走这个实例，拦截器统一处理认证和错误弹窗
- 不需要在每个页面单独判断 Token 过期或请求失败

---

### 3.3 store/user.js — Pinia 状态管理

**核心逻辑**：管理用户登录状态，数据持久化到 localStorage。

**State（状态）**：
- `token`：JWT Token 字符串
- `userId`：当前用户 ID
- `role`：当前角色（resident/worker/admin）
- `userInfo`：用户详细信息对象

**Getters（计算属性）**：
- `isLoggedIn`：检查 token 是否存在且未过期（解析 JWT payload 中的 exp 比对当前时间）

**Actions（方法）**：
- `login(credentials)`：请求 `/api/login`，成功后存储 token/userId/role
- `fetchCurrentUser()`：请求 `/api/users/current`，获取用户详细信息
- `logout()`：清空所有状态，跳转登录页

**持久化**：pinia-plugin-persistedstate 自动将 state 存到 localStorage，key 为 `community-repair-user`，页面刷新不丢失登录状态。

## 四、核心业务流程

### 4.1 完整的工单生命周期

```
居民提交报修 → 工单状态: pending（待派单）
       ↓
管理员派单给维修工 → 工单状态: assigned（已派单），维修工收到通知
       ↓
维修工开始维修 → 工单状态: processing（处理中）
       ↓
维修工完成维修，填写结果 → 工单状态: completed（已完成），居民收到通知
       ↓
居民查看工单并评价 → 评价记录存入 evaluation 表
```

**取消流程**：仅 `pending` 状态的工单可被居民取消，状态变为 `cancelled`。

### 4.2 通知流程

1. **工单提交** → 通知所有管理员
2. **派单** → 通知被分配的维修工
3. **工单完成** → 通知居民
4. **维修工注册** → 通知所有管理员
5. **审核通过/拒绝** → 通知维修工
6. **维修工回复评价** → 通知居民（新增）
7. **密码重置成功** → 通知用户（新增）
8. **管理员发布公告** → 批量通知所有居民和活跃维修工（新增）

### 4.3 权限控制流程

```
用户请求 → JwtInterceptor 解析Token → 存入UserContext
    → Controller 接收请求 → checkRole() 验证角色
    → Service 层进一步验证数据归属权（如"工单是否属于当前用户"）
    → 返回数据
```

**举例**：居民只能取消自己的工单，维修工只能更新自己工单的状态，管理员可以查看所有。

## 五、常见答辩问题速答

**Q1：为什么用 JWT 而不是 Session？**
- JWT 是无状态的，服务器不存 session，适合分布式部署
- Token 包含用户 ID 和角色，服务端解析后就知道用户是谁
- 前端存在 localStorage，刷新页面不会丢失登录状态

**Q2：密码怎么存储的？**
- 用 BCrypt 加密后存数据库，不可逆
- 登录时用 `matches(raw, encoded)` 比对，无需解密

**Q3：怎么防止越权操作？**
- 拦截器验证 Token 有效性
- Controller 用 checkRole() 校验角色
- Service 层校验数据归属（如工单是否属于当前居民）

**Q4：分页怎么实现的？**
- 前端传 page 和 pageSize
- 后端计算 `offset = (page-1) * pageSize`，SQL 用 `LIMIT offset, pageSize`
- 同时执行 COUNT 查询获取总记录数，返回 total 和 rows

**Q5：跨域问题怎么解决的？**
- 后端 WebConfig 配置 CORS 允许所有来源
- 前端 Vite 配置代理，开发环境下 /api 请求转发到 localhost:8080

**Q6：ThreadLocal 为什么要 clear？**
- Tomcat 使用线程池，线程会被复用
- 不清除的话，下一个请求可能拿到上一个用户的 ID，造成数据泄露

**Q7：通知怎么实现的？**
- 后端插入通知记录到 sys_notification 表
- 前端每30秒轮询 `/notifications/unread-count` 获取未读数量
- 点击铃铛时请求 `/notifications` 获取通知列表
- 标记已读后更新 is_read 字段

**Q8：工单紧急标记怎么实现的？**
- `repair_order` 表新增 `priority` 字段（normal/urgent）
- 居民提交报修时可选择"普通"或"紧急"
- 列表查询用 `ORDER BY CASE priority WHEN 'urgent' THEN 0 ELSE 1 END` 确保紧急工单排在前面
- 前端用红色"紧急"标签显示

**Q9：维修工回复评价怎么实现的？**
- `evaluation` 表新增 `reply` 和 `reply_time` 字段
- 新增 `PUT /evaluations/reply` 接口，仅维修工可调用
- Service 层验证维修工是该工单的负责人才能回复
- 回复后自动给居民发送通知

**Q10：公告管理怎么实现的？**
- 新增 `announcement` 表存储公告（标题、内容、发布者、发布时间）
- 管理员在「公告管理」页面可以发布、查看、删除公告
- 发布公告时，使用 `NotificationMapper.batchInsert()` 一次性向所有居民和活跃维修工发送通知
- 居民和维修工在「社区公告」页面可查看所有历史公告，同时在铃铛通知中收到新公告提醒

**Q11：忘记密码功能怎么实现的？**
- 新增 `POST /users/password-reset` 接口
- 用户输入用户名和手机号，后端验证两者匹配
- 匹配则生成随机密码，更新数据库，并发送通知告知新密码
